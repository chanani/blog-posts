얼마 전 새벽, 모니터링 시스템의 알림이 발송되어 확인해보니 운영 중인 사내 사이트에 **초당 10,000건 이상의 API 요청**으로 인해
서버가 다운되는 상황이 발생했습니다. 서비스는 접속이 불가능해졌고, 약 5시간 동안 서비스가 완전히 중단되었습니다.

악의적인 DDoS 공격이었지만, DDoS에 무방비하게 당한 것은 사전에 대비책을 마련하지 않았기 때문이라고 생각합니다.

동일한 상황을 방지하고자 자발적으로 처리율 제한 장치를 구축했고, 이번 글에서는 어떤 알고리즘을 선택했고, 왜 선택했는지에 대해 공유하려고 합니다.

## 📊 API 호출 횟수 제한 정책

처리율 제한 장치를 설계하면서 가장 먼저 고민했던 부분은 임계값이었습니다.
정상 트래픽은 유지하면서, 공격 트래픽을 제한할 수 있는 임계값을 구하기 위해
Google Analytics와 이전에 만들어놓은 모니터링 시스템의 과거 데이터를 분석한 결과,
<strong style="color:#ee2323;">IP별 초당 요청 수를 20건, 1분당 100건으로 제한</strong>했습니다.

**👉🏻 왜 초당 20건인가요 ?**

정상적인 사용자의 경우 초당 평균 2~3건의 요청이 발생했고, 트래픽이 집중되는 시간대에도 초당 10건을 넘는 경우는 없었습니다.
반면 DDoS 공격 당시에는 초당 10,000건 이상의 요청이 발생했습니다. 이 데이터를 바탕으로 **IP별 초당 요청 수를 20건으로 제한**했습니다.

**👉🏻 왜 1분당 100건인가요 ?**

초당 20건 제한만으로는 부족한 경우가 있다고 판단했습니다.
악의적으로 초당 19건씩 지속적으로 요청을 보낸다면 처리율 제한 장치를 우회하면서도 서버에 부담을 줄 수 있기 때문에,
이를 방지하기 위해 1**분 동안 누적 요청이 100건을 초과하는 IP는 자동으로 블랙리스트에 등록**하도록 설정했습니다.

정상 사용자가 1분 동안 발생시키는 요청은 평균 50~60건 수준이었고,
일시적인 트래픽 증가(페이지 새로고침 등)를 고려하여 여유분을 둔 100건으로 임계값을 설정했습니다.

## 😈 Redis를 활용한 블랙리스트 관리

분산 환경에서 여러 서버가 운영되고 있기 때문에, 블랙리스트를 한 곳에서 관리할 필요가 있었고, 적절한 저장소로 **Redis를 선택**했습니다.  
Redis를 선택한 이유는 아래와 같습니다.

### **👉🏻 빠른 응답 속도**

처리율 제한 장치는 모든 요청마다 거쳐야 되기 때문에 성능이 중요

### **👉🏻 TTL 기능**

블랙리스트 IP를 일정 시간 후 자동으로 만료 처리(영구 차단이 아닌 임시 차단으로 정상 사용자 복구 가능성 보장)

### **👉🏻 분산 환경에서의 일관성**

한 서버에서 차단된 IP는 즉시 모든 서버에 반영

**👉🏻 Token Bucket**

<strong style="color:#006dd7;">장점</strong> → IP당 토큰 수와 갱신 시간만 저장하여 메모리에 효율적이며, 정상 사용자의 요청은 허용하고 대량 요청은 토큰 소진으로 차단 가능  
<strong style="color:#ee2323;">단점</strong> → 실제 요청의 타임스탬프를 저장하지 않아, 토큰 잔량으로만 판단

**👉🏻 Sliding Window Log**

<strong style="color:#006dd7;">장점</strong> → 가장 정확한 요청 수 계산 가능  
<strong style="color:#ee2323;">단점</strong> → 모든 요청의 타임스탬프를 Redis에 저장해야 하므로 DDoS 공격 시 메모리 부하 발생

#### 어떤 알고리즘을 선택했나 ?

가장 중요하게 생각했던 부분은 **메모리의 효율성, 정상 트래픽 보호, 공격 트래픽 구분**이었고,
이러한 이유로 **Token Bucket 알고리즘을 선택**했습니다.
Sliding Window Log 알고리즘의 장점도 명확하지만 실제 DDoS 공격 상황에서 메모리 부하로 인해 시스템이 다운될 수 있다고 판단되었습니다.
